# دليل المُراجع — سير الموافقات

**الأدوار:** المُراجع (`reviewer`)، وكذلك الأدوار الإدارية (مدير القبيلة/نائب المدير/مدير الفرع) حين تراجع.
**التطبيق:** لوحة إدارة القبيلة (`admin-web`).

كل تعديل يقترحه دور غير إداريّ (مُساهم/مُشاهد) يصبح **طلب تغيير (Change Request)** يحمل رقعة JSON-Patch (RFC-6902) على الكيان المستهدف. لا شيء يمسّ البيانات الحيّة حتى حالة **«منشور»**.

---

## دورة حياة طلب التغيير

```
مسوّدة → مُقدَّم → قيد المراجعة → (معتمَد | مرفوض | مطلوب تعديل) → منشور
                                   وحالتان نهائيّتان: تعارُض، منتهٍ
```

- عند بلوغ عدد موافقات **«موافق» المتمايزة** الحدَّ المضبوط (`approvalsRequired`) يصبح الطلب **معتمَدًا** ويُنشَر **تلقائيًّا** (تطبيق ذرّي + إعادة فحص التعارُض ← «منشور» أو «تعارُض»).
- **رفض واحد** ⇐ «مرفوض». **طلب تعديل** ⇐ «مطلوب تعديل» (يُعيد صاحب الطلب التحرير والتقديم).
- لا يجوز للمراجع اعتماد طلبه هو، وكلّ مراجع يُحتسَب مرّة واحدة.
- إذا تغيّر `version` للهدف منذ `baseVersion`، ينتقل الطلب إلى **«تعارُض»** ولا يُطبَّق (على صاحبه إعادة الصياغة).

---

## 1. قائمة المراجعة — `/change-requests`

> **وصف الشاشة:** عنوان «قائمة المراجعة»، ومُصفّي **الحالة** (مُقدَّم/قيد المراجعة/مطلوب تعديل/معتمَد/منشور/مرفوض/تعارُض/منتهٍ) مع مسح، وزرّ تحديث. جدول بأعمدة: **الهدف** (شخص/وحدة…)، **العملية** (إنشاء/تحديث…)، **مُقدِّم الطلب** (مُعرّف مختصر)، **الحالة** (وسم ملوّن)، **تاريخ الإنشاء**، وزرّ **مراجعة**. رسالة «لا طلبات» عند الفراغ.
>
> `> [لقطة: قائمة المراجعة — تُضاف عند النشر]`

- المصدر: `GET /change-requests?queue=true&status=` (queue = الطلبات المنتظِرة مراجعتك). الوصول لهذه الصفحة للمراجعين والأدوار الإدارية.

---

## 2. تفاصيل طلب التغيير — `/change-requests/:id`

> **وصف الشاشة:** رأس ببيانات وصفيّة (الهدف، العملية، الحالة، تاريخ الانتهاء). عند حالة «تعارُض» تظهر رسالة حمراء. ثمّ **جدول الفرق (Diff)** بثلاثة أعمدة: **الحقل**، **القيمة القديمة**، **القيمة الجديدة** (والحذف مُبرَز بالأحمر). إن كنت صاحب الطلب وحالته تسمح، يظهر **محرِّر الرقعة** (إطار كهرمانيّ) لتعديل القيم ثم **حفظ كمسوّدة** أو **إعادة تقديم**. إن كنت مراجعًا مؤهَّلًا يظهر **لوح المراجعة**: قائمة **القرار** (موافقة/طلب تعديل/رفض)، حقل **تعليق**، وزرّ **إرسال المراجعة**. وأخيرًا **سجلّ المراجعات** السابقة بوسوم ملوّنة.
>
> `> [لقطة: تفاصيل طلب التغيير — تُضاف عند النشر]`

الإجراءات:
- تحميل الطلب: `GET /change-requests/:id` (يتضمّن `reviews[]`)؛ ولأهداف الأشخاص تُجلَب القيم القديمة عبر `GET /persons/:id`.
- **إرسال المراجعة:** `POST /change-requests/:id/review` بجسم `{ decision: approve|reject|request_changes, comment? }`.
- يظهر لوح المراجعة فقط عندما تكون الحالة **مُقدَّم** أو **قيد المراجعة** ولستَ صاحب الطلب.

---

## 3. طلباتي — `/my-requests`

> **وصف الشاشة:** عنوان «طلباتي» وزرّ تحديث. جدول: **الهدف**، **العملية**، **الحالة** (وسم؛ ومع «مطلوب إجراء» يُظلَّل الصفّ بالكهرمانيّ)، **تاريخ الإنشاء**، وزرّ **تعديل** (عند الحاجة) أو **عرض**.
>
> `> [لقطة: طلباتي — تُضاف عند النشر]`

- المصدر: `GET /change-requests?mine=true`. الطلبات في حالة «مسوّدة/مطلوب تعديل/تعارُض» تحتاج إجراءً منك (تعديل + إعادة تقديم).

---

## إشعارات سير العمل

تصلك إشعارات لحظيّة (وبريد) عند: طلب جديد، اعتماد، رفض، طلب تعديل، واقتراب الانتهاء. النقر على إشعار طلب تغيير من **جرس الإشعارات** ينقلك مباشرةً إلى تفاصيله. كما تُجري مهامّ مجدولة **كنس الانتهاء** (تحويل الطلبات المتجاوزة لتاريخها إلى «منتهٍ») وتنبيهًا قبل الانتهاء بثلاثة أيّام.
