# 04 — الاحتفاظ بالسجلات وسلامة الأدلة

## 1. سجل التدقيق (Audit Log)

يُسجَّل كل تعديل مُقيَّد بالمستأجر مع من/ماذا/متى/IP/قبل-بعد (المواصفة، القسمان 4.5 و10؛ `apps/api/src/modules/audit/audit.service.ts`).

### 1.1 بنية سجل التدقيق (`audit_logs`، تحت RLS)

| الحقل | المصدر | المحتوى |
|---|---|---|
| `tenant_id` | سياق الطلب | المستأجر (تحت RLS) |
| `user_id` | سياق الطلب | الفاعل (من) — **لا من المدخلات** |
| `action` | الخدمة | الفعل، مثل `person.update` |
| `entity_type` / `entity_id` | الخدمة | الكيان المتأثّر (ماذا) |
| `before` | JSON | الحالة قبل التعديل |
| `after` | JSON | الحالة بعد التعديل (فرق JSON) |
| `ip` | سياق الطلب | عنوان المصدر — **لا من المدخلات** |
| `request_id` | سياق الطلب | ربط بالسجل المنظَّم (pino) |
| `created_at` | افتراضي | الطابع الزمني (متى) |

### 1.2 ضمانات السلامة
- **`who/ip/request_id` تُقرأ من سياق الطلب حصراً، لا من مدخلات المستخدم** — فلا يمكن للفاعل انتحال هويته في السجل.
- **الكتابة المفضّلة داخل نفس المعاملة (`recordTx`):** يُكتب سجل التدقيق ضمن معاملة الكتابة نفسها، فيبقى السجل والتغيير متّسقين ذرّياً.
- **التسجيل لا يكسر الطلب أبداً:** في المسار المستقل (`record`) تُلتقط الأخطاء وتُسجَّل دون إفشال العملية — لكن ذلك مقصور على التدقيق خارج معاملات المجال.
- سجل التدقيق **تحت RLS**، فلا يُرى إلا ضمن مستأجره؛ وقراءته مقصورة على `audit.read` (Tribe/Deputy Admin).

### 1.3 نطاق الأحداث المُدقَّقة (فعلي)
من الكود: `person.create/update/delete`، `union.create/update/delete/remarry`، `tribalUnit.create/update/delete`، `changeRequest.create/submit/update/review/published`، `import.upload/submit/publish`، `import_rollback`، `document.create/delete`، `viewRequest.approve/reject`، `visibilitySettings.update`، `workflowSettings.update`، `tenant.settings.update`، `reputation.thresholds.update`، `device.register/deregister`.

## 2. الحذف الناعم (Soft Delete)

- الأشخاص والوثائق تُحذف **ناعماً** عبر عمود `deleted_at` (`persons.deleted_at`, `person_documents.deleted_at`)؛ لا حذف فعلي للبيانات الحسّاسة.
- الفهارس تشمل `(tenant_id, deleted_at)` لاستبعاد المحذوف بكفاءة.
- التصدير والقراءات تُرشِّح `deletedAt: null` (`export.service.ts`).
- **الأثر الرقابي:** الحذف قابل للاسترجاع ويُبقي أثراً؛ يُقترَن دائماً بسجل تدقيق يحمل `before`.

## 3. الإصدار والقفل التفاؤلي (Optimistic Locking)

| العنصر | الآلية | الموضع |
|---|---|---|
| إصدار الشخص | `version Int @default(1)` | `persons.version` |
| أساس طلب التغيير | `base_version` يُلتقط عند الإنشاء | `change_requests.base_version` |
| فحص التعارض عند النشر | إعادة فحص ذرّية؛ إن تغيّر الأساس ⇒ حالة `conflict` بدل النشر | `change-request.service.ts` → `publish` |

يمنع هذا الكتابة فوق تعديل متزامن دون كشف، ويوثّق التعارض بدل تجاوزه صامتاً.

## 4. النسخ الاحتياطي واسترجاع الأدلة

- **الهدف المنصوص عليه (المواصفة، القسم 9):** `pg_dump` يومي + سكربت استرجاع موثّق ومُختبَر؛ واسترجاع قبيلة واحدة ممكن عبر تصدير tenant-scoped.
- **التصدير المُقيَّد بالمستأجر (مطبَّق):** وحدة `exports` تُنتج تصدير الأشخاص (`xlsx`/`csv`) بأعمدة قالب الاستيراد (round-trip)، وتصدير الشجرة (PDF/PNG) — كلها تمر عبر RLS ومُحلِّل الرؤية وتُرشِّح المحذوف (`apps/api/src/modules/exports/`).
- **ملاحظة أمانة:** سكربت `pg_dump` اليومي والاسترجاع المجدوَل هما **إجراء تشغيلي** (هدف §9)، وليسا جزءاً من كود التطبيق؛ يجب توثيقهما وأتمتتهما على مستوى البنية التحتية (راجع `05-business-continuity-and-dr.md`).

## 5. سلامة الأدلة وعدم التلاعب

| الضابط | الوصف | الحالة |
|---|---|---|
| مصدر الهوية الموثوق | الفاعل/IP/request_id من السياق لا المدخلات | مطبَّق |
| ذرّية التدقيق | السجل يُكتب داخل معاملة التغيير | مطبَّق (`recordTx`) |
| ربط عبر request_id | ربط سجل التدقيق بسجل pino المنظَّم | مطبَّق |
| فرق قبل/بعد | لقطة JSON للحالتين | مطبَّق |
| عزل السجل | audit_logs تحت RLS، لكل مستأجر | مطبَّق |
| منع التعديل بأثر رجعي (WORM) | تخزين غير قابل للكتابة/توقيع تشفيري للسجلات | **Backlog** — لا يوجد WORM/توقيع؛ الحماية حالياً منطقية (لا واجهة تعديل/حذف لسجلات التدقيق) لا فيزيائية |
| ختم زمني موثوق خارجي | Timestamping Authority | **Backlog** |
