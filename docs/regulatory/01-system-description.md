# 01 — وصف النظام

## 1. الغرض

منصة TFTSP هي نظام **SaaS متعدد المستأجرين (Multi-Tenant)** يتيح لكل قبيلة أو عائلة بناء شجرة الأنساب الخاصة بها وإدارتها والتحكم في رؤيتها. تُعالِج المنصة بيانات **شخصية وأنساب حسّاسة**، ومن ثمّ فإن العزل الصارم بين المستأجرين هو أخطر متطلب في النظام: تسريب أنساب قبيلة إلى أخرى يُعدّ إخفاقاً أمنياً جوهرياً (المواصفة، القسم 4).

## 2. البيانات المعالَجة وتصنيفها

| فئة البيانات | أمثلة الحقول | الحسّاسية | الموضع |
|---|---|---|---|
| هوية الأشخاص | `full_name, first_name, father_name, grandfather_name, family_name, laqab, gender` | عالية | `persons` (تحت RLS) |
| بيانات ميلاد/وفاة | `birth_date, birth_place, death_date, death_place, is_deceased` | عالية | `persons` |
| روابط النسب | `father_id, mother_id, tribal_unit_id`، جدول الإغلاق `person_closures` | عالية | تحت RLS |
| الزيجات | `unions (husband_id, wife_id, status)` | عالية | تحت RLS |
| الوثائق والصور | `person_documents (object_key)` في MinIO، `photo_key` | عالية | كائنات MinIO + RLS |
| بيانات اعتماد المستخدمين | `password_hash (Argon2id)`, رموز التحديث (تُخزَّن كـ SHA-256) | حرجة | `users`, `refresh_tokens` (مستوى المنصة) |
| سجلات التدقيق | من/ماذا/متى/IP/قبل-بعد | عالية | `audit_logs` (تحت RLS) |
| طلبات المشاهدة | `view_requests` مع مرفق هوية | عالية | تحت RLS |

**بيانات غير مُعالَجة في v1 (خارج النطاق):** بيانات الدفع الإلكتروني، أرقام الهواتف كحقل مستقل في `persons` (مؤجَّل — راجع القرار D-308)، بيانات المصادقة متعددة العوامل.

## 3. الأطراف والأدوار

النظام يعرّف **تسعة أدوار** موزّعة على مستويين:

### مستوى المنصة (خارج نطاق RLS)
- **Super Admin** — مسؤول المنصة الأعلى؛ يمكنه العمل داخل أي مستأجر يختاره صراحةً (عبر `SuperAdminGuard`).
- **Platform Admin** — إدارة تشغيلية على مستوى المنصة.

### مستوى المستأجر (تحت RLS)
- **Tribe Admin** — المسؤول الأعلى للقبيلة.
- **Deputy Admin** — النائب؛ يُنفَّذ بحقول صلاحية زمنية لا بدور خاص.
- **Branch Admin** — مقيّد بوحدة `tribal_unit` محددة وما تحتها.
- **Reviewer** — مراجِع/معتمِد لطلبات التغيير.
- **Contributor** — مساهم في البيانات عبر طلبات التغيير.
- **Viewer** — مُشاهِد؛ قد يُمنح صلاحية مؤقتة عبر طلب مشاهدة.
- **Guest** — غير عضو (سياق طلبات المشاهدة العامة).

تُخزَّن الأدوار في `role_assignments (user_id, role, tribal_unit_id?, valid_from, valid_to, anchor_person_id?)` — وهي بيانات تفويض مقصودة خارج RLS ليتمكن جدار الحماية من قراءتها بفلترة صريحة `(tenant_id, user_id)`.

## 4. المكوّنات المعمارية

| المكوّن | التقنية | الدور |
|---|---|---|
| الواجهة الخلفية (API) | NestJS + Prisma | منطق الأعمال، التفويض، فرض العزل | 
| قاعدة البيانات | PostgreSQL 16 | تخزين مع RLS مفعّل على كل الجداول tenant-scoped |
| التخزين الكائني | MinIO | الوثائق والصور، عبر روابط موقّعة قصيرة الأجل (15 دقيقة) |
| الطابور والمهام | Redis + BullMQ | مهام إدارية (مثل إعادة بناء جدول الإغلاق) |
| البريد | SMTP (MailHog في التطوير) | التنبيهات عبر البريد |
| الإشعارات الفورية | FCM | إشعارات تطبيق الموبايل (M5) |
| واجهات الويب | Angular (`admin-web`, `platform-web`) | لوحات الإدارة |

## 5. مستويا قاعدة البيانات (Two Planes)

يفصل النظام بوضوح بين مستويي وصول للبيانات (راجع `apps/api/src/common/prisma/prisma.service.ts`):

1. **مستوى المستأجر (`tenant`)** — يتصل بدور قاعدة بيانات `tftsp_app` الذي **لا يملك BYPASSRLS**؛ كل قراءة/كتابة مُقيَّدة تلقائياً بالمستأجر عبر امتداد Prisma الذي ينفّذ `SET LOCAL app.current_tenant`. هذا هو المسار الوحيد لمعالجة بيانات المستأجرين.
2. **مستوى المنصة (`platform`)** — يتصل بدور المالك (يتجاوز RLS غير المُجبَر) ويُستخدم **حصراً** بواسطة طبقة المصادقة/المنصة الموثوقة (خلف `SuperAdminGuard` أو تدفّق ما قبل المستأجر) للقراءات عبر المستأجرين وجداول المنصة (`tenants`, `users`, `role_assignments`, `tenant_subscriptions`).

## 6. التدفّقات الرئيسية

### 6.1 تدفّق المصادقة
تسجيل الدخول → التحقق من كلمة المرور (Argon2id) → فحص القفل → إصدار JWT وصول (15 دقيقة) + رمز تحديث (30 يوماً). يحمل الـ JWT `tenantId` المُصادَق عليه، وهو **المصدر الوحيد** لهوية المستأجر في كل طلب لاحق.

### 6.2 تدفّق قراءة بيانات شخص
الطلب → استخراج `tenant_id` من الـ JWT → `SET LOCAL app.current_tenant` → استعلام مُقيَّد بـ RLS → **مُحلِّل الرؤية (Visibility Resolver)** يُزيل الأشخاص خارج النطاق (404) ويحجب الحقول الممنوعة → الاستجابة.

### 6.3 تدفّق تعديل البيانات (سير الموافقات M2)
مساهم يُنشئ طلب تغيير → مراجِعون (غير المُنشئ) يعتمدون → عند بلوغ نصاب الموافقات المطلوب يُطبَّق التغيير ذرّياً مع فحص التعارض → تسجيل تدقيق.

### 6.4 تدفّق رفع الوثائق
طلب رابط رفع موقّع (رفض SVG بالنوع المُعلن) → الرفع المباشر إلى MinIO → التأكيد يجلب البايتات الفعلية ويفحص البصمة الرقمية (Magic Bytes)، ويرفض SVG/غير المدعوم، ويفرض سقف 10 ميغابايت على الحجم الحقيقي.

## 7. بيئة التشغيل

تُوصَّف البنية في `docker-compose.yml`: خدمات `postgres:16-alpine`، `redis:7-alpine`، `minio`، `mailhog`، وخدمة `api` (NestJS). يتصل الـ API بقاعدة البيانات عبر متغيّرين: `DATABASE_URL` (دور `tftsp_app` بلا BYPASSRLS للتشغيل) و`DATABASE_MIGRATION_URL` (دور المالك `tftsp` للهجرات فقط) — راجع القرار D-005.

## 8. حدود النطاق (Backlog صريح)

وفق المواصفة (القسم 3) والقرارات، العناصر التالية **غير مبنية في v1**: SMS/WhatsApp/Telegram، ميزات الذكاء الاصطناعي، المصادقة متعددة العوامل (MFA)، بوابات الدفع الإلكترونية، تصدير GEDCOM، استيراد ODS، الشجرة الشعاعية/الزمنية، البيومتري، والروابط العميقة.
