# DECISIONS.md — ambiguity resolutions (Spec Section 11.9)

Every choice not spelled out by the spec is logged here (simplest viable option), per the agent behaviour rule.

## Foundation (Lead/PM)
- **D-001 Monorepo tooling:** npm workspaces (not Nx/Turbo) for M1 — simplest that satisfies the `apps/*` + `packages/*` layout. Revisit if build orchestration grows.
- **D-002 `packages/shared-ui` deferred:** In M1 each Angular app keeps its own components; a shared component library is extracted later to avoid two agents contending on one package. The `admin-web` and `platform-web` apps each own their UI for now.
- **D-003 Two separate Angular apps** (`apps/admin-web`, `apps/platform-web`) rather than one Angular workspace with two projects — keeps agent workstreams in disjoint directories and matches the "separate domains/deployments" intent of the spec.
- **D-004 API contract as the sync point:** `docs/API_CONTRACT.M1.md` + `packages/shared-types` are frozen before web work fans out. Backend owns changes and announces them.
- **D-005 DB roles:** migrations run as owner (`tftsp`), the app connects as `tftsp_app` WITHOUT BYPASSRLS so RLS is always enforced (Spec Section 4.2).
- **D-006 Partial dates:** year-only birth/death stored as `YYYY-01-01` with a `*_precision` marker deferred to when the UI needs it; API accepts `YYYY` or full ISO.

## Integration / reconciliation (Lead/PM, post-agent)
- **D-300 Backend is source of truth for wire shape.** After all three agents landed, reconciled cross-agent mismatches to the backend's actual implementation: (a) persons list envelope `{ data, page, pageSize, total }` — flipped admin-web from `items`→`data`; (b) conflict `messageKey`s use backend snake_case (`errors.person.duplicate_candidates` / `version_conflict`) — flipped admin-web's two constants; (c) `POST /platform/tenants` body is camelCase (`nameAr`/`nameEn`, matches backend DTO + shared-types) — flipped platform-web from snake_case (supersedes the earlier platform-web D-102). Both web apps rebuilt clean after the flips.
- **D-301 `docker-compose` api service wired** by the Lead using the backend's ready Dockerfile (owner runs `prisma migrate deploy`, app connects as `tftsp_app`); container-internal hostnames override the localhost `.env` defaults.
- **D-302 CI** (`.github/workflows/ci.yml`): api lint+build+unit+e2e (Testcontainers isolation/lineage/auth gates) + both web builds. Flutter jobs deferred to M5.
- **D-303 Tenant-settings endpoint is an open M1 backend gap** — admin-web's settings page assumes `GET/PATCH /tenant/settings`; backend did not build it. Tracked in the contract as TODO; dispatched to the Backend agent as a focused follow-up.

## platform-web (Platform-Web agent, M1)
- **D-100 Angular 21 (not 22):** the environment's Node is `v22.22.2`; the Angular 22 CLI hard-requires `>=22.22.3` and refuses to run. Angular 21.2 (previous LTS, supports `^22.12`) is used instead — still "latest usable LTS" here. Bump to 22 once Node is updated.
- **D-101 Zoneless change detection:** the Angular 21 scaffold ships zoneless by default (no `zone.js`). Kept it — matches the Signals-first mandate. All view state uses signals; templates read signals (incl. `LanguageService.isRtl()`) so they re-render reactively.
- **D-102 Create-tenant request body casing:** `POST /platform/tenants` is sent with **snake_case** `{ name_ar, name_en, slug, admin }` exactly as written in `docs/API_CONTRACT.M1.md` (the frozen sync point), even though `packages/shared-types` `CreateTenantDto` is camelCase. The contract doc is treated as authoritative for the wire shape. If Backend actually expects camelCase, flip `CreateTenantRequest` in `core/models/tenant.model.ts`.
- **D-103 Tenant list-response shape is casing-tolerant:** `GET /platform/tenants` count fields are unspecified. `PlatformService` normalizes each row from any of `personsCount` / `persons_count` / `counts.persons` (same for users) and either `nameAr` or `name_ar`, so the UI is robust to whichever the Backend emits.
- **D-104 Token storage in `localStorage`:** simplest persistence that survives reload for a desktop-only ops console; silent refresh + chain-revocation on reuse is handled by the interceptor. Revisit (httpOnly cookies) only if XSS surface grows.
- **D-105 Dev API access via proxy:** `apiBaseUrl` is the relative `/api/v1`; `proxy.conf.json` forwards `/api` to `http://localhost:3000` in dev and Nginx handles it in prod — no per-env rebuild of the URL.
- **D-106 Tailwind v4 + PrimeNG Aura, dark "ops console" shell:** platform-web deliberately uses a dark slate/indigo chrome (distinct from admin-web) to reinforce the separate security boundary; the routed content sits on a light canvas for readability.
- **D-107 No ESLint in-app yet:** the Angular 21 scaffold no longer adds ESLint by default; repo-wide lint/CI is left to the Lead rather than pinning a lint stack unilaterally in this sub-app.

- **D-206 Minimal SVG tree only:** `/tree` view is a basic depth-layered vertical SVG of nodes/edges (mirrors with language). The rich d3/canvas renderer with LOD + lazy generation expansion (Spec §7) is deferred to M4 per M1 admin-web scope.
- **D-207 Pagination envelope assumed:** API_CONTRACT says `/persons` is "paginated" without a shape; assumed `{ items, total, page, pageSize }` (`Paginated<T>` in `common.ts`). Adjust the mapping in `PersonService.list` if Backend uses a different envelope.

## Backend M1 (Backend agent)
- **D-101 `role_assignments` kept OUT of RLS:** it carries `tenant_id` (+ a `(tenant_id, user_id)` composite index per §4.1) but is authorization metadata that must be resolvable BEFORE a tenant is bound (login must list a user's memberships to pick the active tenant). It is only ever read through the auth layer / `PolicyGuard`. Every genuinely tenant-scoped domain table (persons, unions, person_closures, tribal_units, audit_logs) IS under RLS.
- **D-102 Two DB planes:** the request/tenant plane uses role `tftsp_app` (NOBYPASSRLS) via a Prisma client extension that runs `SET LOCAL app.current_tenant` per transaction; a trusted platform plane (auth membership resolution + Super-Admin platform aggregates) uses the owner role, which bypasses non-forced RLS. RLS is enabled but NOT `FORCE`d, so the owner can do cross-tenant counts for `/platform/stats`; the app role can never escape its tenant.
- **D-103 Closure table is PATERNAL (`father_id` edges):** models the نسب genealogy tree, keeps `depth` unique (a tree, not a DAG), and powers `/tree`, `/ancestors`, `/descendants`. `mother_id` is stored and gender-validated but not part of the closure/tree in M1 (maternal-line views are a later concern). Self-ancestry is prevented via a closure lookup before any `father_id` change; a `mother_id` set to self is also rejected.
- **D-104 Soft-delete detaches:** soft-deleting a person re-parents its direct children to roots (`father_id = NULL`, via the same closure move routine) and drops the person's closure rows, keeping the closure consistent without hard-deleting descendants.
- **D-105 Refresh tokens:** high-entropy opaque strings; only a SHA-256 hash is stored. Rotation links tokens by `family_id`; presenting an already-rotated (revoked) token revokes the whole family (reuse detection).
- **D-106 Partial dates:** `YYYY` is accepted and stored as `YYYY-01-01` UTC (matches Lead D-006); full ISO passes through. A precision marker is deferred.
- **D-107 Tenant create body is camelCase** (`nameAr`/`nameEn`) to match `packages/shared-types` `CreateTenantDto`; `docs/API_CONTRACT.M1.md` updated accordingly (Backend owns the contract).
- **D-108 Tribe-settings logo presign stubbed in M1:** `POST /api/v1/tenant/settings/logo-upload` returns the agreed `{ uploadUrl, logoKey }` shape (logoKey = `tenants/<tenantId>/logo-<uuid>`, 15-min intended TTL) so admin-web can integrate now, but the URL is a best-effort placeholder — real MinIO presigning + magic-byte validation is M4 scope (Spec §3/M4). Shape is stable; only the signing internals change later. `GET/PATCH /api/v1/tenant/settings` are full M1 (Tenant is platform-level/outside RLS, so the service scopes every read/write explicitly by the JWT tenantId — there is no tenant id in the path/body, so a Tribe Admin cannot target another tribe).

## admin-web (Admin-Web agent, M2)
- **D-208 Change-requests list shape:** `GET /change-requests` is assumed to return a plain `ChangeRequest[]` (the M2 contract says "list; filters" without a pagination envelope, unlike `/persons` and `/notifications`). If Backend paginates it, wrap the return in `ChangeRequestService.list` and read `.data`.
- **D-209 Non-admin person write = create + auto-submit:** when a non-admin saves the person form, admin-web `POST /change-requests` (draft) then immediately `POST /:id/submit`, so the proposal lands in the review queue in one action (the user clicked "Submit for review"). Further edits/resubmits happen from **My Requests → detail**. Direct admin write is unchanged.
- **D-210 Socket.IO wiring:** namespace `/notifications`, connected **same-origin** (`socketUrl:''`) — the dev-server proxies `/socket.io` with `ws:true` to :3000 and Nginx does so in prod (no CORS). The JWT is sent in the handshake `auth` as a **function** so a rotated token is re-read on every (re)connect. The socket lifecycle lives entirely in `NotificationService`; the authenticated `ShellComponent` calls `init()` on entry and `reset()` on logout. No token-refresh interception on the socket — it simply reconnects with a fresh token.
- **D-211 Requester display:** `ChangeRequest.createdBy` is a user id only (no display name in the contract), so the queue shows a truncated id. Swap to a name if Backend adds a `requester` object.
- **D-212 Owner patch editor:** the detail page edits a change request via a **generic per-field editor** (typed text/number/checkbox derived from each patch op's value type) rather than a raw-JSON textarea — friendly for person fields; other target types still edit their raw values. Save = `PATCH /change-requests/:id { patch }`, Resubmit = PATCH then `/submit`.
- **D-213 No duplicate pre-check on the CR path:** duplicate detection (Spec §8, threshold 0.6) is a `POST /persons` (direct write) concern; the non-admin change-request create path does not surface duplicate candidates in the UI (the M2 contract doesn't define them for `/change-requests`). Assumed the backend re-checks at publish time.
- **D-214 Form reachability for non-admins:** the persons list "Add"/edit now route everyone to the person form (non-admins get the proposal flow). **Delete stays admin-only** — deletion-as-change-request is not in the M2 admin-web UI scope.

## Backend M2 (Backend agent)
- **D-201 Publisher reuses M1 write paths:** auto-publish applies the JSON patch by calling the M1 `*InTx` methods (`PersonsService.createInTx/updateInTx/softDeleteInTx`) and the tribal-unit/union `*Tx` repo methods inside ONE `tenantTransaction`, so lineage/closure + audit stay identical to direct writes and the apply + CR-status update are atomic (Spec §3 M2 "atomic apply").
- **D-202 Conflict re-check granularity:** `baseVersion` conflict detection is version-based for `person` (the only versioned M1 entity). For `union`/`tribal_unit` (no `version` column) `baseVersion` is null and conflict is existence-based (target deleted since creation ⇒ conflict). Matches shared-types (`baseVersion?`) and the person-focused DoD gate. Adding versions to union/tribal_unit is deferred.
- **D-203 Notifications are created server-side via the owner client** with an explicit tenantId (never user input), so the BullMQ jobs (no HTTP/tenant context) can create them too; reads/mark-read use the RLS app client scoped to the current user. `notifications`, `change_requests`, `change_request_reviews`, `workflow_settings` are all tenant-scoped and under RLS.
- **D-204 Socket.IO auth:** the `/notifications` namespace authenticates on handshake from `auth.token` (preferred), `Authorization: Bearer`, or `?token=` — the same ACCESS JWT. The socket joins room `t:<tenantId>:u:<userId>`; the server emits `notification` (payload = full Notification row) to that room. No tenant/user is trusted from the client beyond the verified token.
- **D-205 Email channel:** one bilingual (ar RTL + en LTR) MJML template rendered per notification type, subjects/bodies from the `notifications` i18n namespace, sent over SMTP (MailHog in dev) via the `NotificationChannel` abstraction. Delivery is best-effort — SMTP/render failures are caught and never break the request. New channels (SMS/WhatsApp — Backlog) plug into the same interface.
- **D-206 Scheduler gating:** the BullMQ `JobsModule` (Redis-dependent) is only imported when `ENABLE_SCHEDULER !== 'false'`. The maintenance logic lives in a Redis-free `ChangeRequestMaintenanceService` (always loaded) so the expiry sweep/warning can be unit/e2e-driven without Redis; the processor/scheduler just call it. Expiry sweep runs hourly, the expiring-soon warning daily (3-day window).
- **D-207 State-change notifications:** submit → all reviewers (minus author) get `change_request_submitted`; reject → owner `change_request_rejected`; request_changes → owner `change_request_changes_requested`; quorum reached → owner `change_request_approved` then `change_request_published` (or `change_request_conflict`); expiry job → `change_request_expired` / `change_request_expiring`. `under_review` (partial approvals) is a non-terminal wait state and does not notify.

## admin-web (Admin-Web agent, M2.5)
- **D-215 Import list/rows envelope:** `GET /imports` and `GET /imports/:id/rows` are read as `{ data, page, pageSize, total }` (reusing `Paginated<T>`), matching the M2.5 contract. `ImportService` returns them as-is.
- **D-216 `/imports` socket reuses the M2 pattern:** same handshake as `/notifications` (JWT in `auth` as a function, same-origin via the `/socket.io` ws proxy). All wiring lives in `ImportService` (`connect()`/`disconnect()` + `progress$` Subject). The wizard connects on upload and disconnects on submit/destroy; the batch-detail connects only while the batch is in a processing status. Progress events are matched to the active batch by `importBatchId`.
- **D-217 Ambiguous-ref resolution via person pickers:** the `ImportRow` model carries no candidate list for `ambiguous` rows, so the preview resolves them with `/persons` autocomplete pickers (gender-filtered client-side, per D-204) that `PATCH resolvedFatherId`/`resolvedMotherId`. Spouse resolution is omitted from the table for now (father/mother cover the common case); the DTO/endpoint still support `resolvedSpouseId` if a later need appears.
- **D-218 Duplicate merge target defaulting:** for `duplicate_candidate` rows the UI PATCHes `mergeTargetId` only when the reviewer explicitly picks a person; when `decision=merge` is chosen without a pick, the backend is expected to default the merge target to the row's `duplicateOfId`.
- **D-219 Preview paginates per row-status:** errors, duplicates, and ambiguous rows are three independent lazy PrimeNG tables, each driven by `?status=&page=` (honest to the single-status filter). Error rows are flattened client-side to one `(row, column, message)` line per error within the current page; column labels come from `imports.columns.*`, messages from the backend `messageKey`.
- **D-220 Plan-limit / magic-byte checks are server-side:** the wizard only does a **client-side 50 MB size hint** and file-type `accept` filtering; the authoritative 50 MB + magic-byte + subscription/`maxPersons` checks run on the backend (M2.5 contract note), and their `messageKey` is surfaced as an error toast on upload.
- **D-221 Rollback refusal shape:** a refused rollback is assumed to return `ApiErrorBody` with `messageKey` + `details.dependencies: string[]`; the detail page renders the dependency list under the error. Adjust the `details` key if Backend differs.
- **D-222 Import gating:** the Import nav entry is shown to `canWrite` roles (Tribe Admin / Deputy / Branch Admin); the batch **Rollback** action is gated to **Tribe Admin** (Spec §12). The backend enforces both regardless of UI gating.

## Backend M2.5 (Backend agent)
- **D-301 Plan-limit stand-in:** subscription management is M4, so M2.5 enforces a per-tenant `Tenant.max_persons` (default Free = 500) checked at submit BEFORE the batch enters the workflow/apply. Over-limit ⇒ 400 `errors.import.plan_limit_exceeded` with `{ current, max, available, requested }`. Swap for the real plan in M4 without touching the guard call site.
- **D-302 Batch = one Change Request (targetType `import_batch`):** the batch is submitted as ONE M2 CR (operation `create`, targetId = batchId). The M2 `ChangeRequestService` delegates publish/reject of `import_batch` CRs to the imports module via an optional `IMPORT_BATCH_APPLIER` token (mutual `forwardRef`), so M2 stays independent of M2.5. Publish is NOT the single-tx generic path — the applier runs chunked transactions (1,000 rows/tx, Spec §12) with in-file parents created before children (topological order).
- **D-303 import_batch_id tags CREATED persons only:** merges update a pre-existing person through the M2 JSON-Patch update path but do NOT overwrite its `import_batch_id` — otherwise rollback would soft-delete a record the batch didn't create. Rollback therefore soft-deletes only batch-created persons; merges are not reverted.
- **D-304 Worker plane:** the parse pipeline (`ImportParseService`) and progress writes use the owner client with an explicit tenant_id (no HTTP/tenant context in the BullMQ worker), mirroring the M2 jobs (D-203). Request-time reads/writes use the RLS app client. Parsing is Redis-free and gated exactly like the M2 scheduler: `ImportDispatcher` enqueues to BullMQ when `ENABLE_SCHEDULER!=='false'` (JobsModule registers the real enqueue fn on init); tests call `ImportParseService.run` directly.
- **D-305 Streaming + magic bytes:** uploads are streamed to MinIO with busboy (never fully buffered); xlsx is verified by ZIP magic bytes (`PK\x03\x04`), CSV by extension; 50 MB hard limit. exceljs streaming WorkbookReader (xlsx) + csv-parse (CSV) keep parse memory bounded for 100k rows.
- **D-306 Closure rebuild on rollback runs inline** in the (tenant-context) rollback request via `LineageService.rebuildClosure` (recursive father-walk rebuild), not a separate BullMQ job — the job would lack tenant context and rollback must complete for its audit/response. Can be offloaded for very large tribes later.
- **D-307 Reference resolution:** in-file `ref:<rowRef>` (pass 1) vs DB resolution by exact `name_normalized` (+ clan when given) (pass 2); >1 DB match ⇒ `ambiguous` (manual pick via PATCH). Duplicate detection stays fuzzy (pg_trgm similarity ≥ 0.6, §8). Spouse references are resolved/validated but union creation from import is deferred (persons-only apply in M2.5).

## admin-web (Admin-Web agent, M3)
- **D-223 Public route is guard-free and shell-free:** the tree-view request form is mounted at top-level routes `/request-view` and `/t/:tenantSlug/request-view` (siblings of `/login`, before the `**` catch-all), so it renders without the authed shell, `authGuard`, notification socket, or bell. Language + RTL still work (the app-level i18n initializer runs regardless).
- **D-224 Public POST uses SKIP_AUTH:** `ViewRequestService.createPublic` / `uploadIdAttachment` set the interceptor's `SKIP_AUTH` HttpContext so no bearer is attached and no silent-refresh runs on the public route (even if a logged-in admin happens to open it).
- **D-225 ID-attachment upload endpoint assumed:** the M3 contract only guarantees a "minimal key handoff". The public form does an optional two-step upload — `POST /view-requests/id-attachment` (multipart, public) assumed to return `{ idAttachmentKey }`, then `POST /view-requests` with that key. The attachment is always optional in the UI; the backend enforces `requireIdForViewRequest` and its error `messageKey` is surfaced. If Backend presigns instead, only `uploadIdAttachment` changes.
- **D-226 tenantSlug source:** `/t/:tenantSlug/request-view` binds the slug from the URL (field hidden); `/request-view` shows a "tribe code" field. Requiredness is validated in `submit()` (not via reactive validators) to avoid input-binding timing races with `withComponentInputBinding`.
- **D-227 Approve expiry = ISO datetime:** the approve dialog sends `validTo` as a full ISO string from a `p-datepicker`, with client-side guard rejecting empty/past dates (`minDate=today`); backend remains authoritative on expiry.
- **D-228 View-request list assumed a plain array:** `GET /view-requests?status=` is read as `ViewRequest[]` (no pagination envelope specified). The Tribe-Admin pending badge in the shell is derived from `list('pending').length`, fetched once on shell init (not live — refreshed on navigation into the page).
- **D-229 ID-attachment viewing:** the review table shows only a "paperclip / ID attached" indicator, not a working download link, because no attachment-download/presign endpoint is defined for M3 (that is M4 document handling). Wire a presigned link when M4 lands.
- **D-230 Redaction tolerance:** person/tree components now treat redacted (absent, not null) fields defensively — tree nodes default `name/gender/isDeceased`, the persons list guards `gender`, and the person form coalesces `gender/isDeceased` on load. No component dereferences an optional person field without a guard.

## Backend M3 (Backend agent)
- **D-308 Redactable Person fields in M1 schema:** the resolver drops `photoKey` (when `showPhotos=false`) and `birthDate` (when `showBirthDates=false`) — the only sensitive columns the M1 Person has. `showPhones`/`showDocuments` map to columns that don't exist until M4 (phone, documents); the policy flags are stored now and wired to those fields when they land. Existence policies (`showDeceased`, `showMinors`, women-hidden) remove the whole person (→ 404 / absent), field policies DELETE the key (never null).
- **D-309 Member scope anchoring:** `role_assignments` gains `member_scope` (direct|clan|branch|tribe, null ⇒ tenant `defaultMemberScope`) and `anchor_person_id` (the member's own person node). `clan`/`branch` scope = the tribal-unit subtree rooted at the assignment's `tribal_unit_id` (closure of the unit tree); `direct` = the anchor's direct ancestors (closure) + children + siblings + self. Admin roles (tribe/deputy) = tribe scope + bypass; `branch_admin` = its unit subtree + bypass. `admin` baseline level blocks all non-admins.
- **D-310 Women-hidden semantics:** `womenDisplay='hidden'` shows women ONLY to admins and to a viewer's direct relatives; the other three modes (under_father/with_siblings/under_husband) are tree-PLACEMENT hints for the client and do not hide data in M3. An external Viewer (no anchor) therefore sees no women.
- **D-311 Existence hiding = 404:** anything outside the viewer's visibility (scope, women-hidden, deceased/minor policy) returns 404 — never 403 — so a caller can't infer a person exists. Applied at `/persons/:id`, `/tree` root, `/persons/:id/ancestors|descendants`, and per-row in list/search/tree/lineage.
- **D-312 Resolver routing:** the ONE resolver is injected into `PersonsService` (list/search/findOne) and `LineageService` (tree/ancestors/descendants). Viewer context is built OUTSIDE the tenant transaction (its own queries need the wrapped RLS client); filtering is pure and runs inside. M1/M2/M2.5 admin write & preview flows return full data to their admin/author actors and are not person-read echoes, so they don't route through the resolver.
- **D-313 Temporary grants:** view-request approval creates a fresh Viewer `User` + a `role_assignments` row with `valid_to` (no new role type — reuses M1 validity window). A random credential is set; a real login/onboarding flow for granted viewers is deferred (M4). Expiry is enforced in `PolicyGuard`: no active assignment + an expired one for the tenant ⇒ 401 `errors.auth.grant_expired` (distinct from 403).
- **D-314 Public view-requests use the owner client:** submission is public (no JWT/tenant context), so the row is created via the owner client with an explicit tenantId resolved from `tenantSlug`; admin notifications (new `NotificationType.view_request_submitted`, added to schema + shared-types) are persisted the same way. Admin list/approve/reject run authenticated under RLS.
- **D-315 Public ID-attachment upload:** `POST /view-requests/id-attachment` (@Public, field `file` + `tenantSlug`) streams to MinIO under `view-request-ids/<tenantSlug|uuid>/<uuid>` and returns `{ idAttachmentKey }`. Accepts PNG/JPEG/WEBP/GIF + PDF by MAGIC BYTES (not extension); **SVG (and any leading markup/XML) is rejected** — the §M4.3 XSS rule applied early; 10 MB cap. `tenantSlug` is required to scope the key. NOTE: no dedicated rate-limiter exists yet in the codebase — a per-IP throttle on this public route (and `POST /view-requests`) is a follow-up (M4); for now abuse is bounded only by the 10 MB cap + single-file limit.

## platform-web (Platform-Web agent, M4)
- **D-400 Manual activation = PUT with note (no separate activate flag):** `SetSubscriptionDto` has only `{ tier, expiresAt?, note? }`, so "manual bank-transfer activation" is modeled as the PUT itself; the note field is labelled as the transfer reference and lands in the activation log. No extra `activate` boolean invented.
- **D-401 Expiry alerts sourced from `PlatformDashboard.expiringSoon`:** `GET /platform/tenants` (list) carries no subscription expiry, so the tenants list makes one best-effort `GET /platform/stats/dashboard` call and builds a `tenantId → expiresAt` map from `expiringSoon` to drive the per-row "expires in Nd" badge + amber row highlight. If the dashboard call fails, the list still works (no badges).
- **D-402 By-plan chart is a dependency-free CSS bar chart:** rather than add chart.js / `p-chart`, the "tribes by plan" breakdown is plain HTML/CSS horizontal bars. Colors use the dataviz-skill validated categorical palette (blue `#2a78d6`, aqua `#1baf7a`, yellow `#eda100`, green `#008300`); each bar carries a direct count + tier-name label, which satisfies the palette's relief rule for the two sub-3:1 slots. Identity is never color-alone.
- **D-403 New `/statistics` route; M1 `/dashboard` kept:** the richer `PlatformDashboard` lives at a new `/statistics` route + nav item. The original basic `/dashboard` (M1 `GET /platform/stats`) is left intact rather than replaced, to avoid disturbing merged M1 behaviour.
- **D-404 `GET subscription` 404 = empty state, not an error:** a tribe with no subscription yet returns 404; the manager treats that as "no plan" (defaults the picker to Free) instead of showing an error banner. Other statuses surface a load error.
- **D-405 Expiry input casing:** the expiry date is entered via a native `<input type=date>` and sent as a full ISO string (`new Date(value).toISOString()`); the tier picker surfaces `PLAN_LIMITS` caps (`≤ 500`, `∞` for enterprise) inline so the operator sees the person cap when choosing a tier.
- **D-406 Subscription UI as a per-row dialog (not a separate detail route):** kept subscription management in the tenants list via a `SubscriptionManagerComponent` dialog opened per row — matches "add a subscription section/tab per tribe" without introducing a tenant-detail routing layer in M4 scope.

## admin-web (Admin-Web agent, M4)

The compact `{nodes,edges}` feed is turned into a strict hierarchy using the **father edge as structural parent (mother edge fallback)**; multiple roots hang off a hidden virtual super-root. Layouts: vertical, horizontal, and **fan (radial d3.tree)**. **≤1500 visible nodes → SVG; above → Canvas with LOD** (labels drawn only when zoom scale > 0.5). Pan/zoom via `d3-zoom`; **lazy expand** fetches +2 generations via `GET /tree(nodeId, 2)` and merges; **search** centers the camera and highlights the ancestor path; **"path between"** mode picks two nodes and highlights A→LCA→B; **RTL** mirrors the x axis (and the radial angle for fan). ## Backend M4 (Backend agent)
- **D-401 Plan cap superseded:** the person cap now derives from the tenant's subscription tier (`PLAN_LIMITS`, Free=500…Enterprise=unlimited) via a central `PlanLimitService.assertCanAddPersons`, enforced on direct person-create, import submit, and single-person contribution publish. The M2.5 `Tenant.max_persons` column is left in place (harmless) but no longer consulted; over-cap ⇒ **403** `errors.subscription.plan_limit_reached` `{ tier, max, current }`. The M2.5 import e2e was updated to the new mechanism (seed to 499, import 3 → 403).
- **D-402 Subscriptions are platform-level** (like Tenant, OUTSIDE RLS): `tenant_subscriptions` (one per tenant) + `subscription_activations` (log), managed only via `SuperAdminGuard` under `/platform/tenants/:id/subscription`. Manual (bank-transfer) activation goes through a `PaymentGateway` abstraction (`ManualPaymentGateway`) — the activation row IS the platform-level audit trail (tenant-scoped AuditLog can't be written without a tenant context).
- **D-403 Documents:** presign (15-min PUT) → client PUTs to MinIO → confirm. Confirm re-reads the ACTUAL first bytes for a magic-byte check (shared `detectFileKind`) — SVG/markup rejected outright (§M4.3 XSS), 10 MB cap enforced on the real object size (not the client's declared size). GET returns 15-min presigned download URLs (never public). Every path passes the M3 resolver (out-of-scope person → 404).
- **D-404 Crowdsourcing reuses the M2 engine:** a contribution is a Change Request with a `contribution_type` column — no parallel workflow. Create-time gating: pending cap (`ReputationThresholds.maxPending`, default 20) → 403 `too_many_pending`; viewers limited to `edit_data`/`add_source` and only when `allowViewerContributions` → else 403 `viewer_not_allowed`; person target run through the resolver → out-of-scope 404. On the review decision, published contributions bump `accepted`, rejected bump `rejected`, `accuracyRate` recomputed immediately and `trustLevel` re-derived from per-tenant thresholds. **No automatic privilege promotion** (v1).
- **D-405 Stats via materialized views:** `tribe_stats_mv` (per-tenant person aggregates) + `platform_dashboard_mv` (platform totals) refreshed hourly by a BullMQ job (`StatsRefreshService`, Redis-free so tests/on-demand call it directly) + `POST /stats/refresh`. Matviews carry no RLS, so they're read via the owner client with an EXPLICIT tenant filter; fast-changing counts (units, pending CRs, contributors, generations, byGeneration) are computed live and combined.
- **D-406 Puppeteer-in-CI:** exports use `puppeteer-core` (never downloads Chromium; `PUPPETEER_SKIP_DOWNLOAD=true` on install). `PdfRenderer.resolveExecutable()` picks `PUPPETEER_EXECUTABLE_PATH` → `PLAYWRIGHT_BROWSERS_PATH/chromium` (default `/opt/pw-browsers/chromium`) → common system paths; if none, exports return `errors.export.failed`. The RTL/paper/structure of the render input (`buildTreeHtml`) is unit-tested, so CI verifies correctness without launching a browser; the xlsx/csv exports need no browser.
- **D-407 Biography sanitization:** the `add_biography` contribution type writes `Person.biography`, a rich-text field wired into `CreatePersonDto`/`UpdatePersonDto` (JSON-Patch `/biography`, ≤20k chars). Every write path (direct create/update AND the change-request publisher, which shares `createInTx`/`updateInTx`) runs the value through `sanitizeBiography` (allow-list of formatting tags via `sanitize-html`; scripts/handlers/iframes/`javascript:` stripped; links forced `rel="noopener noreferrer nofollow"`). Sanitization lives in the service (not the DTO) so contributions applied via the publisher — which casts patch fields and bypasses class-validator — are still sanitized. Unit-tested in `sanitize-html.spec.ts`.

## Backend M5 (Backend agent)
- **D-501 Device registration is tenant-scoped under RLS:** `device_registrations` (migration `0007_device_registrations`) carries `tenant_id` + a `tenant_isolation` RLS policy like every M3/M4 tenant table. The FCM `token` is **globally unique** (a physical device holds one token), so re-registration **upserts by token** — refreshing owner/tenant/platform/lastSeenAt rather than duplicating. `POST /devices` returns 200 (upsert, not a fresh create → not 201); `DELETE /devices/:token` is owner-scoped (a member only drops their own token) and idempotent (`{ removed: 0|1 }`). Endpoint gated by a new `device.manage` permission mapped to all member roles (READ_ROLES) = "any authenticated member". Register/deregister are audited (`device.register`/`device.deregister`); token pruning is treated as system hygiene (logged at debug, not audited).
- **D-502 FCM is the third NotificationChannel, disabled-by-default:** `FcmNotificationChannel` joins in-app + email in `NOTIFICATION_CHANNELS`, so any created Notification (M2 CR state changes, M3 view-requests) also pushes. `FcmService` reads `FCM_PROJECT_ID`/`FCM_CLIENT_EMAIL`/`FCM_PRIVATE_KEY`; if **any** is absent (dev/CI) it is permanently disabled and every call is a safe no-op — **never throws on bootstrap or send** (the DoD requirement). firebase-admin is lazily `require`d only on first real send behind a dedicated app name (`tftsp-fcm`); an init/send failure disables push and is swallowed. The data block carries the notification `type` + JSON `payload` (+ `notificationId`) for deep-open on tap; title/body reuse the existing `notifications.<type>.{subject,body}` i18n keys (no new keys). Tokens FCM reports as `registration-token-not-registered` / invalid are pruned from `device_registrations`. NotificationsModule imports DevicesModule to reuse `DeviceRepository` (one-way dependency, no cycle).
